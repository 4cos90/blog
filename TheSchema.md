## 0. 目标与边界

> [!info] 核心目标
> 维护本 Blog 的内容资产，以**可复用、有条理**的方式管理文章和素材

- **目标**：管理 `raw/` 素材库，支持 Blog 文章的写作与发布
- **边界**：
  - 原始素材（`raw/`）是灵感来源，整理后可用于写作参考
  - 文章发布到 `_posts/` 目录
  - 不要动其它目录

---

## 1. 目录结构约定

### 素材层
- `raw/`：原始素材（网页 Markdown、灵感片段、笔记等），按时间命名

### Blog 内容层
- `_posts/`：发布的 Blog 文章，每篇文章一个文件
- `wiki/`：辅助知识库（如需整理临时笔记）

### 根目录文件
- `wiki/index.md`：素材索引（可选）
- `wiki/log.md`：操作日志

---

## 2. 页面类型与基本格式

### 2.1 Blog 文章

路径：`_posts/YYYY-MM-DD_标题.md`

Frontmatter：
```yaml
---
title: "文章标题"
date: "YYYY-MM-DD"
tags: ["tag1", "tag2"]
summary: "一句话说明文章核心观点"
---
```

正文使用 Markdown，保持用户朴实直接的文风。

### 2.2 素材页

路径：`raw/YYYYMMDD_HHmmss_标题.md`

来源素材，按原样保存，可包含：
- 网页抓取的 Markdown 内容
- 用户直接发送的笔记片段
- 灵感、想法、引用等

## 3. 工作流：Ingest / Write / Lint

### 3.1 Ingest（导入素材）

当用户发送 URL 或素材内容时：

1. **URL**：使用 `WebFetch` 抓取，转换为 Markdown，保存到 `raw/`
2. **MD 内容**：直接保存到 `raw/`
3. 在 `wiki/log.md` 追加记录：
   ```
   ## [YYYY-MM-DD] ingest | raw/xxx.md
   ```

### 3.2 Write（写作协助）

当用户请求写作或改写时：

1. **确认需求**：用户想要写什么？（新文章 / 改写已有内容 / 续写）
2. **参考素材**：如有相关素材在 `raw/`，阅读并提取要点
3. **参考文风**：阅读 `raw/` 下已有文章，学习用户用词习惯
4. **起草/改写**：保持朴实直接、结构清晰、个人视角的风格
5. **输出位置**：保存到 `_posts/YYYY-MM-DD_标题.md`
6. 记录到 `wiki/log.md`：
   ```
   ## [YYYY-MM-DD] write | _posts/xxx.md
   ```

### 3.3 Lint（健康检查）

当用户说"检查一下 blog"时：

1. 扫描 `_posts/`，检查：
   - 文章是否有 frontmatter
   - 标签是否规范
   - 内容是否有明显问题（断句、逻辑）
2. 生成建议清单，**不直接修改**
3. 经用户确认后再执行

## 4. 约定与风格

### 4.1 文风要求

> [!tip] 写作原则
> 保持朴实、直接、有温度的风格

- **朴实直接**：避免过度修辞，用简单的话说明复杂的事
- **结构清晰**：善用 `#` 标题层级、bullet points
- **有具体数字**：少用"很多"、"不少"，多用"76%"、"4小时"
- **有个人视角**："我"的体验、"我的"观点
- **结尾升华**：适当总结提炼，但不过度煽情

### 4.2 命名约定

- 文件名：中文 + 下划线，稳定可读
- 标签：英文小写，用连字符分隔（如 `ai`、`vibe-coding`）
- 标题：直击要点，避免标题党
