# 为什么我认为 skill 不是一个很好的范式

## 引言

在用 AI Agent 干活的时候，我们很容易沉浸在「它能做什么」里，却忽略了一个更基本的问题：「它这样做安全吗？」

Skill 是现在主流的 agent 扩展方式，概念很简单——给 agent 装上各种技能包，让它能调用外部能力完成更复杂的任务。但我自己用下来，遇到过几个让我特别难受的场景，这些经历让我对 Skill 这个做法产生了挺大的疑虑。

---

## 案例一：电脑莫名其妙多了个 .NET 环境

事情是这样的：我工作中需要偶尔操作 Word 文档，用的是 minimax-word 这个 skill。

在没装这个 skill 之前，Claude Code 遇到要创建 docx 的情况，会自己写个 Python 脚本——能跑，好使。

装了 skill 之后，同样的需求，它就不走 Python 那条路了，改调用 skill，用 C# 来创建文档。问题来了：我电脑上压根没有 .NET 开发环境。于是 Claude Code 二话不说，自己去装了 .NET SDK——静默的、闭环的、没问过我一声的。

这里有两个让我不舒服的点：

**第一，原来的方案被降权了。** Python 脚本明明能工作，但因为 skill 的存在，agent 觉得「既然有这个 skill 就用这个吧」。这对于我自己来说还好，但对于不懂技术的用户，他只会觉得：为什么今天电脑变慢了？为什么多了个看不懂的东西？

**第二，agent 在我不知情的情况下改了我的电脑。** 装了 5GB 的 .NET SDK，就为了创建一次 Word 文档。下次遇到类似需求，可能又装一套别的。几次下来，电脑里就乱成一锅粥了。

---

## 案例二：邮件操作能不能封装成 skill？

第二个案例是一次用户需求：用户用 ChatFlow 生成了一个静态 HTML 页面，想把它塞到 Outlook 邮件正文中（不是附件，是正文里）。

当时我想，Outlook 对 PowerShell 支持挺好的，要不把邮件操作封装成一个 skill 吧——读取、搜索、发送、插入 HTML，功能很合理。

但我停下来想了一会儿：如果把 PowerShell 脚本封装进 skill，那就得给 Claude Code 执行 bash 的权限。那理论上，它也可以生成这样一条命令：

```powershell
Remove-Item -Recurse -Force $HOME
```

用户的邮件、文档、所有东西，一行命令全删干净。

有人说：那就不给通用 bash 权限，只开放固定的几个脚本。但这里有个根本问题——agent 的本事是组合和推理，你限制的是「命令叫什么名字」，而不是「命令能产生什么效果」。用规则去对抗智能，规则迟早会被绕过去。

最后我没做成 skill，而是用本地服务的方式实现了邮件的读取和搜索能力。删除和修改的接口压根不存在，agent 调不到。

---

## Skill 的三个根本问题

这两个事情让我想明白了一件事：Skill 的设计，让 agent 的自主权太大了。

**第一，运行时它说了算，环境说改就改。**

Skill 执行依赖本地环境，agent 觉得需要什么运行时就装什么。这个过程对用户完全黑盒，但结果是用户的电脑被悄悄改了。技术出身的人还能诊断一下，非技术用户就只能干瞪眼。

**第二，bash 权限一给出去，就收不回来了。**

Skill 需要 bash 权限，但 bash 权限本身就是管理员级别的。你没法在给权限的同时又精确限制「只能做 A，不能做 B」——这是一个架构层面的矛盾，不是配几个文件能解决的。

**第三，agent 做了什么，用户完全不知道。**

Agent 为什么选了这个 skill 而不是那个？为什么放弃了原本能跑通的方案？用户只看到结果，不知道过程。这种黑盒在关键业务场景里是很危险的。

---

## 我的当前选择

我不是说我找到了最佳实践，实话实说。在邮件那个场景里，我的做法是：本地起一个邮件处理服务，只暴露读和搜索的接口，删除和修改的接口根本不做。

这个方案不优雅，甚至可能不是行业以后会走的路。但它让我不需要回答一个问题：我愿不愿意把系统权限交给这个 agent？

在企业场景里，我们的用户往往不是技术出身的。他们不会排查问题、不会判断异常、不知道 agent 在他们电脑上干了什么。Skill 给他们的「自主权」，对他们来说不是赋能，是负担。

---

## 写在最后

 Skill 的想法其实挺美好的——agent 自由调用各种能力，闭环把事情做完，多省心。但在把它交给非技术用户之前，我觉得我们得先想清楚一件事：

我们真的准备好承担 agent 自主决策的后果了吗？

至少现在，我的答案是否定的。
